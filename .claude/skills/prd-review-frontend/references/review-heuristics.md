# PRD 评审启发式清单（前端视角）

## 0) 严重级别口径（建议）

- `P0 Blocker`：缺失关键信息/依赖，无法评估或无法实现；或高概率导致事故/合规风险
- `P1 Major`：可实现但风险高，容易返工/体验崩；或会显著影响核心指标
- `P2 Minor`：有替代方案或影响较小，但建议补齐以提升质量
- `P3 Nit`：一致性/文案/规范类小问题

## 1) PRD 最容易缺的关键信息

- 目标/指标：成功指标是否可量化（转化率、留存、时长、失败率、性能指标）？
- 范围边界：明确 in-scope / out-of-scope，避免“顺手加上”导致无限膨胀。
- 用户与场景：用户分层、权限差异、灰度人群、地区/语言/时区差异。
- 依赖与时间线：后端、设计、数据、权限、法务/隐私、第三方 SDK。
- 失败策略：服务不可用/弱网/超时/重试/降级时体验与提示语。
- 数据口径：字段含义、枚举值、时间语义（UTC/本地）、精度、单位。

## 2) 用户旅程与“状态矩阵”

对每个页面/模块，至少要能回答：
- 入口：从哪里来（深链/站内跳转/推送/分享）？是否需要登录？
- 展示：首屏信息是什么？可用性优先级？是否需要 skeleton？
- 交互：编辑/提交/撤销/批量/快捷键/拖拽等是否明确？
- 退出：返回/关闭/离开提醒（未保存）？

状态矩阵（常缺）：
- `loading`：首次加载/刷新/分页/搜索中
- `empty`：无数据、无权限、条件筛选导致为空（提示文案不同）
- `error`：网络错误/业务错误/部分成功（哪些可重试）
- `submitting`：重复提交防抖、并发操作冲突
- `offline`：是否需要离线可用或明确“不支持”

## 3) 前端可实现性与交互细节

- 响应式：断点、列表密度、触控目标尺寸、横竖屏（如涉及）
- 可访问性：键盘可达、焦点管理、语义标签、对比度、读屏文案
- i18n：文本长度膨胀、复数、日期/货币/数字格式、RTL（如可能）
- 表单：校验规则、错误提示位置、输入法、自动填充、提交失败回显

## 4) API / 数据契约（常见踩坑点）

最少要有：
- 请求/响应示例（含错误示例）
- 字段定义：类型、是否可空、默认值、枚举全集
- 分页：`cursor` vs `page`，排序稳定性，重复/缺失数据如何处理
- 搜索/过滤：是否支持模糊、大小写、分词、拼音等；防抖建议
- 业务错误：错误码列表 + 用户可读文案策略（前端/后端谁负责）

高风险信号：
- “前端自己算/自己拼”关键业务字段（口径漂移）
- 模糊的时间字段（createdAt/updatedAt 是 UTC 还是本地？）
- 隐私字段返回但不应展示/存储（PII、token、手机号等）

## 5) 非功能性评审（高级前端工程师关注点）

- 性能预算：首屏/交互延迟目标、包体上限、图片策略（格式/懒加载）
- 可靠性：重试/退避、幂等、乐观更新冲突、缓存一致性、回滚路径
- 安全：XSS/CSRF、token 存储、权限校验（不要只靠前端隐藏）
- 观测：错误上报（含上下文）、关键路径埋点、Web Vitals、告警阈值

## 6) 平台与兼容性（经常被遗漏）

- 平台：Web/H5/Hybrid；是否需要小程序/内嵌 WebView 兼容
- 浏览器/系统：最低版本、是否支持 IE/旧安卓、是否需要无障碍模式
- 网络：弱网/离线是否需要支持；图片/视频大资源策略
- 输入：键盘/触控/手写/输入法；横竖屏与安全区（刘海屏）

## 7) 评审输出的“可执行粒度”

对每条问题，建议至少包含：
- 结论：缺什么/不清楚什么/哪里有风险
- 影响：用户体验、研发成本、上线风险、数据不可用等
- 建议：明确的补充内容、交互稿要求、API 字段/错误码建议
- 责任方：PM / Design / Backend / FE / Data

## 8) 典型“追问”问题库（可直接复用）

- 权限：哪些角色可见/可操作？被禁止时 UI 怎么提示？
- 竞态：同时编辑/重复点击/多端同步如何处理？
- 灰度：是否需要 feature flag？失败回滚策略是什么？
- 埋点：成功/失败/取消/曝光分别埋哪些事件？字段口径是什么？
- 国际化：是否需要多语言？默认语言和兜底策略是什么？
- 审计：是否需要操作日志？展示给谁？保留多久？
